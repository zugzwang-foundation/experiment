## §3 — The changes

### Change 1 — `drizzle-kit check` in `ci.yml` *(per-PR; offline)*
Add a step (after Install, before `Apply database migrations`): `pnpm drizzle-kit check`. Validates the committed journal/snapshots for collisions/corruption — exactly what a migration-adding PR risks. No DB, no secret, no pg_cron interaction. The job-level `env.DATABASE_URL` is already set (the offline check still loads `drizzle.config.ts`, whose `dbCredentials.url` is a non-null assertion — it must be defined even though no connection is made). Fails the job (and, once required, the merge) on a corrupt journal.

### Change 2 — `db:check-drift` in `ci.yml` *(per-PR; CI-safe timestamp+count)*
Add a step **after** `Apply database migrations`: `pnpm db:check-drift`, with `DATABASE_URL` = the existing CI Postgres service URL (the job-level env). Asserts journal head == applied head (catches a #5769-class "migrate exited 0 but stopped short" even in CI). Method is **unchanged** (timestamp+count) — the pg_cron strip removes statements but not journal entries, so `created_at`/count still match. Reuses `scripts/check-migration-drift.ts` verbatim; no `DATABASE_URL_PROD`/`_STAGING` in CI, so it falls through to plain `DATABASE_URL`.

### Change 3 — env-audit (re-scoped) *(OD-1/OD-2/OD-3/OD-4 — all LOCKED)*
New `scripts/ci-env-parity.ts`, structured **pure-logic + thin I/O**:
- **Pure** `auditEnvParity({ vercelKeysByScope, dopplerKeysByConfig, syncs, scopeToConfig, intentionalManual, requiredKeys })` → `{ orphans[], unhealthySyncs[], duplicateSyncs[], missingRequired[] }`. Topology mapping: **Production→prd**, **Preview→stg**, **Staging(custom env)→stg**. **LOCKED scope (a)+(c)+(d)+(b):**
  - **(a) orphans** — Vercel keys present in a scope but absent from that scope's mapped Doppler config, **minus** the Sentry intentional-manual allow-list (reused from `vercel-env-audit.ts`'s `INTENTIONAL_MANUAL`).
  - **(c) unhealthySyncs** — any sync whose status ≠ `In-Sync`.
  - **(d) duplicateSyncs** — more than one sync targeting the same (Doppler config → Vercel scope) destination.
  - **(b) missingRequired** — every required key must exist in each Doppler config; allow-list = `{ DATABASE_URL, BETTER_AUTH_URL, R2_BUCKET_PFP, R2_BUCKET_UPLOADS }` (small explicit must-exist list, **not** a full manifest; `R2_BUCKET_UPLOADS` read from `src/server/storage/r2.ts:45`).
- **I/O clients:** Vercel REST (`GET /v9/projects/{id}/env`) via `VERCEL_API_TOKEN` + project/team IDs; Doppler REST (config secrets + **syncs** list) via `DOPPLER_AUDIT_TOKEN`. **No secret *values* logged** (metadata/keys only — the write-only discipline of `vercel-env-audit.ts`, hardened by the mandatory `@code-reviewer`). Fail-**closed** on any API error (treat as audit failure, exit non-zero).
- **Placement (OD-2 LOCKED):** a dedicated scheduled `.github/workflows/env-audit.yml` (`schedule:` daily + `workflow_dispatch`), failing on **any** finding; **not** a per-PR `ci.yml` step. Rationale (recorded as ADR-0024 Patch P1, note 2): env drift is operator-caused, not PR-caused — gating code review on it couples unrelated failure domains and injects external-API flakiness + token-rotation risk into the merge path.
- **File (OD-3 LOCKED):** new file; `vercel-env-audit.ts` untouched; **ADR-0024 file-map** in-place-patched to name `ci-env-parity.ts` (Patch P1, note 1), same commit.
- **Secrets (OD-4 LOCKED):** operator-provisioned — `VERCEL_API_TOKEN` (read-only) + Vercel project/team IDs; `DOPPLER_AUDIT_TOKEN` (read-only service token: read configs stg+prd + list syncs); referenced by name only.
- **Review (WEB ADDITION 1):** `@code-reviewer` on `ci-env-parity.ts` is **MANDATORY** — focus: no secret-value logging anywhere, tight token handling, fail-closed on API error.

### Change 4 — staging-migrate GHA job *(ADR-0024 item 3; INERT until D3)*
New `.github/workflows/staging-migrate.yml`: `on: push: branches:[staging]`; checkout → setup pnpm/node (`.nvmrc`) → `pnpm install --frozen-lockfile` → install Doppler CLI → `doppler run --config stg -- pnpm db:migrate:staging` (token `DOPPLER_TOKEN_STG`). The `migrate-staging.ts` guard (`DATABASE_URL_STAGING` + `STAGING_PROJECT_REF_FRAGMENT`) is satisfied by Doppler-injected vars. **No pg_cron strip** (staging Supabase has the extension). **Header comment:** "INERT until D3 creates the `staging` branch + repoints `branchMatcher`; until then no push to `staging` occurs and this workflow never fires. Do NOT create the branch here (D3)."

## §7 — Failure modes
- **env-audit token expiry / API outage** → blocks only the **scheduled** `env-audit` run, never a code PR (OD-2 LOCKED to R2 / scheduled, off the merge path). Fail-closed is intentional: a broken audit must not read green.
- **`db:check-drift` false signal in CI** → impossible by construction (timestamp+count, strip-invariant: the strip drops statements, not journal entries).
- **staging-migrate fires prematurely** → impossible until D3 (no `staging` branch); header comment documents it.
- **`drizzle-kit check` needs `DATABASE_URL`** (config non-null assertion) → CI job-level env sets it; offline check otherwise (no connection).
- **Secret-value leak via logs** → guarded by the no-value-logging discipline + the mandatory `@code-reviewer` pass.
