## Context (why this exists)

Production is **serving `ba484f0`** (current `main` HEAD, #174) but its database is **13 migrations behind** (applied through `0005`; journal head `0018`). The two halves drifted apart for a concrete reason:

- Until the recon/verification session, Vercel `autoAssignCustomDomains` was still **`true`** (recon found it `true`; the D3 toggle never actually stuck). So `main` pushes — including #173/#174 (`ba484f0`) — **auto-served** to `zugzwangworld.com` under the *old* behavior, with **no migrate gate**. That is how prod came to serve `ba484f0` against a `0005` DB.
- `autoAssignCustomDomains` was set to **`false`** (SET-3, fresh-GET-confirmed, persisted) — which finally **arms** the migrate-before-serve gate for *future* pushes. The currently-live `ba484f0` alias was not stranded (V5 re-verify: still serving `ba484f0` post-toggle).

So the drift is **purely DB-side**, and the verify-before-serve promote path (runbook §3) has **never been exercised** (`ba484f0` arrived by auto-serve, not by the gated promote).

**This stratum does two things, no more:**
- **D5** — remediate the prod DB drift: apply migrations `0006→0018` to the already-live `ba484f0` build, behind an empty-prod safety gate; verify `/api/health` flips `migrations: drift → ok`.
- **D6** — exercise the now-armed merge → staged → `/api/health` gate → promote → live-verify pipeline **once**, using the runbook-§3-finalize commit as the vehicle; record the promotion note. "Done = documented AND exercised once."

**Preconditions — all green (re-confirmed read-only this session):** main `ba484f0` clean/in-sync; prod serving `ba484f0`; `/api/health` = `{env:prod, db:ok, migrations:drift, canary:ba484f0}`; Doppler `prd` has `DATABASE_URL_PROD` + `PROD_PROJECT_REF_FRAGMENT`; guard passes (prod URL contains prod ref `zbvprdcyxhlguxbostdj`); prod migrate URL is the `:5432` session pooler matching staging; `autoAssignCustomDomains=false` (persisted). `migrate-prod.ts` will no longer refuse.

## Approach (one paragraph)

Run ADR-0022's per-migration-transaction applier (`pnpm db:migrate:prod`) over the 13-migration delta against the live prod DB, but **only after** a read-only empty-prod predicate passes (the two DROP ops `0017`/`0018` would silently lose data if rows existed; the three NOT-NULL ops `0006`/`0014`/`0016` would fail-closed but we catch them up front). Confirm success by the **authoritative** signal — `/api/health` per-hash `migrations: ok` (migrate exit codes are not trusted, drizzle-orm #5769) — plus `db:check-drift`. Then drive **one** real pass of the gated promote pipeline by merging the runbook-§3 finalize commit to `main` (now stages-without-serving), gating on the **staged build's** `/api/health`, promoting, verifying live, and recording the promotion note. **Zero new `src/` code** is written — the per-hash drift detector and the `VERCEL_GIT_COMMIT_SHA` canary already shipped (D1); the serve-gate is configuration (auto-assign OFF, done) + procedure (runbook §3) + the existing `/api/health` surface.

**Reversibility:** `0017`/`0018` are irreversible (column/table/type drops). Justified: both target **deprecated DEBATE.8/9 objects** that are **empty on prod** and already absent from the `ba484f0` schema model — there is nothing to preserve. ### The empty-prod safety predicate (re-checked at apply-time, NOT assumed from snapshot)

Run **before** `db:migrate:prod`, read-only, against prod:

```bash
doppler run --config prd -- node -e '
const postgres = require("postgres");
const sql = postgres(process.env.DATABASE_URL_PROD, { max: 1, prepare: false });
// STOP set — the SIX tables a destructive/contract op touches; ANY non-zero here = ABORT.
const STOP = ["image_uploads","resolution_events","payout_events","mod_actions","comments","friendly_fire_events"];
// INFO set — printed for sanity only. Legitimate serving residue is EXPECTED here (prod has been
// live-serving ba484f0: events accrues every minute via close-due-markets; users/system_state may be
// non-zero). NOT an abort trigger — investigate only if surprisingly large.
const INFO = ["bets","positions","dharma_ledger","markets","pools","users","events","system_state"];
(async () => {
  const blocked = [];
  for (const t of STOP) { const r = await sql.unsafe(`select count(*)::int c from ${t}`); console.log("STOP", t, r[0].c); if (r[0].c !== 0) blocked.push(t); }
  for (const t of INFO) { const r = await sql.unsafe(`select count(*)::int c from ${t}`); console.log("info", t, r[0].c); }
  await sql.end();
  console.log(blocked.length ? `ABORT — non-empty destructive-target table(s): ${blocked}` : "OK — all six destructive-target tables empty; safe to migrate");
  process.exit(blocked.length ? 1 : 0);
})();'
# STOP iff any of the SIX destructive/contract-target tables is non-zero (data-safety).
# INFO rows are sanity only: a non-zero events/users/system_state is EXPECTED serving residue
# (prod has been live), NOT a corruption anomaly — do not reflexively abort or override.
```

(Inline `node -e` under `doppler run` — read-only `SELECT count(*)`, inlines its own `postgres()` client per AGENTS.md §7, no committed script, no raw `psql` against prod. **Narrowed per `@db-migration-reviewer` (LOW-1/SURPRISE):** an equality check on `events`/`users` would false-abort on legitimate serving residue, so the STOP gates only the six tables a destructive/contract op actually touches.)

## 5. Failure modes

| Failure | Detect | Recover |
|---|---|---|
| Empty-prod predicate finds rows in a STOP table | the count query | **STOP** — do not migrate; surface to operator (an unexpected non-empty destructive-target table is a real anomaly). |
| Migration fails mid-run | `migrate-prod` exits 1; per-migration-tx rolls back the failing file | earlier migrations are committed; investigate, fix-forward, re-run (resumes from DB head). |
| `0007` `CREATE EXTENSION pg_cron` not permitted on prod | `0007` tx fails | staging precedent says it works (same Supabase setup); if it fails, `0006` is committed, fix-forward. |
| `/api/health` `migrations` = `error` (not `ok`/`drift`) | the gate curl | the per-hash detector returned `error` (DB unreachable **or** migration files not bundled — D1 `next.config.ts outputFileTracingIncludes`). Treat `error` as **not-ok** → do not promote. (Currently moot: live prod returns a real `drift` verdict, so bundling is intact in `ba484f0`.) |
| Gate not `ok` on the staged build | step-4 curl | **do not promote**; the live alias keeps serving the prior build; fix + re-run. |
| Regression discovered post-promote | live `/api/health` / behavior | re-promote the prior known-good deployment (instant alias swap). Caveat: forward-applied schema is **not** auto-rolled-back — expand/contract is what lets the prior build tolerate the migrated schema. |
| pg_cron jobs (0007/0011) + Vercel cron `close-due-markets` (every minute) hit transitional schema during migrate | Sentry | pre-launch empty + per-table ACCESS-EXCLUSIVE locks are instant on empty tables; the delta does not `ALTER markets` (the cron's table); negligible. Post-migrate, latent drift-errors in the prod crons (if any) resolve. |

## 6. Edge cases

- **A STOP-table not actually empty** → predicate STOPs before any DDL.
- **A migration already partially applied / high-water-mark skip** → `migrate-prod` is head-watermark based (skips files ≤ DB head `created_at`); prod is contiguous `0000–0005` so no gap, and the per-hash `/api/health` gate backstops any #5769 skip regardless.
- **`main` ahead of the serving SHA with prod-unsafe pending changes** → verify `main == ba484f0 == prod canary` before migrate (re-confirmed; no migrations beyond 0018 pending).
- **The promotion-note / log commit itself needs deploying** → with auto-assign OFF, doc commits *stage* without serving; they need **not** each be promoted — they ride the next real promote. Only the D6 exercise promotes once.
- **Two drift surfaces disagree** → by design: `/api/health` per-hash (deployed, unstripped) is the promote authority; `check-drift` timestamp+count is the CI/CLI signal.
