## Context

ADR-0024 closes the deploy-drift **class**. D1 stopped the bleeding (preview→staging DB re-source, per-hash `/api/health`, `prepare:false`, the staging DB migrated to head `0018`). D2 made **CI a required `ci` check** on `main` and landed the three ADR-mandated checks (`drizzle-kit check`, `db:check-drift`, the scheduled env-audit) plus the **inert** `staging-migrate.yml` (ADR-0024 item 3) — explicitly *inert until D3* because no `staging` branch exists, so a push to `staging` never occurs and the workflow never fires.

**D3 un-shadows the staging environment and isolates the prod auto-serve.** Three coupled infra moves, in a strict order whose ordering *is* the safety:

1. **Land the minimal D3 PR** (this plan + the runbook) — deliberately the **last `main` push that auto-serves Production**, merged *before* the auto-assign-domains toggle.
2. **Repoint the Vercel staging `branchMatcher`** `main → staging` (operator), then **create the `staging` branch** off the D3-SHA (CC, `git push origin main:staging`). This single push (a) fires the now-armed `staging-migrate.yml` and (b) matches the now-armed matcher → Vercel auto-deploys staging. Staging becomes a real git-auto-deploy sandbox (ADR-0024 item 2).
3. **Disable Production "Auto-assign Custom Production Domains"** (operator) so future `main` pushes yield staged builds that do not auto-serve — the precondition for the D5 staged-prod-promote. Proven non-disruptive by an R2 before/after `/api/health` curl.

D3 does **not** touch prod migrations (D5), does **not** edit `smoke-staging.ts` (deferred — OD-1 / SURPRISE-1), and does **not** reopen ADR-0024.

Runbook authorship split: **CC** = staging-auto-deploy mechanics (push → `staging-migrate.yml` → Vercel auto-deploy; the `/api/health` curl verification; resettable-sandbox/seed posture; no-pg_cron-strip note; the verified operator toggle navigation below). **Web PLACEHOLDER stub** = the prescriptive prod-promote sequence (ADR-0024 item 5), clearly flagged "to be finalized at D5", pointing to ADR-0024 item 5 — **no fabricated prod-promote prescriptive content**.

### Phase B — arm staging (operator FIRST, then CC)
4. **RELAY TO OPERATOR:** repoint the Staging custom env `branchMatcher` `main → staging` (Vercel → Settings → Environments → [Staging custom env] → Branch Tracking → change match from `main` to `staging`; match rule — `staging` need not exist yet). **Wait for operator confirm.**
5. **Pre-flight** (before the first staging auto-deploy): confirm `BETTER_AUTH_URL` is present in Doppler `stg` (→ synced to Vercel Staging; it fails at **build** time if missing, and the first auto-deploy is a fresh build); confirm `DOPPLER_TOKEN_STG` can `doppler run --config stg` (provisioned D2, never exercised; read-only is fine). Then `git push origin main:staging` (creates `staging` @ «D3-SHA»). This single push (a) fires `staging-migrate.yml` → migrates the staging DB (idempotent; already @ `0018` → 0 new, exit 0); (b) matches the now-armed matcher → Vercel auto-deploys staging.

### Phase C — verify staging GREEN
6. Watch the **staging-migrate** Actions run → **GREEN** (the first real validation of the Doppler CLI action).
7. `curl https://staging.zugzwangworld.com/api/health` → `env:"staging"`, `db:"ok"`, `migrations:"ok"`, `canary == «D3-SHA»`. **[PRIMARY GATE — independent of smoke.]**

## §7 — Failure modes + HALT conditions
- **R2-AFTER canary ≠ «D3-SHA» (stranded alias)** → **HALT, surface, do not proceed.** Reversible: operator re-enables "Auto-assign Custom Production Domains".
- **`staging-migrate` goes RED** → **HALT, surface, do not proceed.** Reversible: the staging DB is a resettable sandbox (drop schema + re-migrate + re-seed).
- **`BETTER_AUTH_URL` missing in Doppler `stg`** → the first staging auto-deploy fails at **build** time → fix the Doppler secret before re-pushing (pre-flight catches it).
- **`staging` branch pushed before the operator repoint** → matcher still `main` → staging does not auto-deploy. Mitigated by the strict order: CC pushes `staging` **only after** the operator confirms the repoint (Phase B step 4 → 5).
- **A `main` merge during the freeze** → would auto-serve before the toggle. Mitigated by freezing `main` at A.3 (no merges until Phase D done); the `chore/d3-log` PR merges **after** Phase D, when auto-assign is already OFF (so it does not auto-serve).
