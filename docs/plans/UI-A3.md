**Per-invariant failure-mode narratives (critical-path law — what ships broken if the assertion is missing or wrong):**

- **INV-1 (the comment-free-buy narrative).** The corruption scenario is a UI affordance that decouples the pair: e.g. a "quick bet" submit enabled while the argument is empty (leaning on the server 400 as UX), or a draft-save path that posts the comment text anywhere except the place body. Both would either train users into a rejection loop or — worse, under a future refactor that "helpfully" splits the calls — place a bet without its argument. The design forecloses it: ONE submit path, one payload builder, gate-on-argument-required; the payload unit test pins that no composer state emits a bet without a non-empty trimmed body. Mandatory commentary is the thesis — a composer that weakens it ships the anti-thesis.
- **INV-2 (the faucet-adjacent narrative).** The composer renders `spendableToday` (balance + unpaid Daily Credit preview). The corruption scenario: treating that preview as spendable via any path that doesn't RUN the accrual — it is only real because the place path pays the credit BEFORE the F-BET-4 check. If a future edit bounded the input by `balance` instead, users would be wrongly blocked; if it bounded by `spendableToday` but the server stopped paying first, over-stakes would 400. The parity is already pinned server-side (A2); the A3 gating unit pins the client bound to the SAME field, and the F-BET-4 400 (belt) maps to a legible state rather than a crash.
- **INV-3 (the side-mutation narrative).** The corruption scenario: a composer whose side chip toggles YES↔NO after open, or a reply composer that derives side from the COLUMN it renders in (it renders OPPOSITE the post — deriving from the slot would invert every reply-bet). A Counter on a NO post is a YES bet rendered in the NO-side slot: slot ≠ side, permanently. The derivation matrix unit pins Support/Counter × parent-side → wire side; the immutability unit pins that no state transition rewrites it. ## 5. Failure modes

- **Double-submit / double-execution** — submit disabled in flight + held key; a network-dropped committed place re-submitted under the held key replays the ORIGINAL 200 (receipts); an edit-after-invisible-commit lands on the F-2 protective 409 (refresh, no auto-resubmit — never a second bet). Detect: lifecycle unit + the two replay/F-2 integration tests. The §1 I-IDEM narrative is the failure story.
- **Poisoned key** — foreclosed by the F-1/F-2-corrected lifecycle: fresh key after a 429 (cached 24h), fresh key on edit-after-terminal, held key ONLY across uncached transients; pinned by the lifecycle matrix (429 row asserts held-key-after-429 ABSENT).
- **Quote/execution drift** — advisory by design (cpmm §6.3; W-1 recompute authoritative); success renders the EXECUTED figures from the response (`sharesBought`/`dharmaReturned`), never the last preview. No tolerance-abort exists by design (Option A).
- **Quote route errors mid-typing** — preview degrades to `—` (no state escalation; the preview is advisory); submit stays governed by gating, not by quote availability.
- **Stale viewer context** (concurrent action in another tab: position sold, balance spent) — bounds/gating go stale until refresh; the server 400s map to legible states; `router.refresh()` on every success re-syncs. Accepted (display-grade reads — A2 law).
- **Moderation 503 hammering** — manual retry only (no auto-retry loop against a fail-closed vendor); held key makes eventual success replay-safe (the 503 is never cached).
- **RSC refresh race** (success → refresh → the new post not yet in view) — single-writer DB read-after-commit renders it; the refreshed model is the same tx-committed state (no eventual consistency layer exists).
- **Client decimal arithmetic drift** (sell conversion) — decimal.js exact strings, capped ≤ quantity; the server pre-check + storage CHECK backstop; conversion unit pins dust/boundary cases.
- **Component regression on the read surface** — DebateView/DebateColumn surgery could break the DEBATE.4 render; mitigation: additive composition (new components; existing ones edited minimally), full suite + manual smoke gate.
- **Stale `.next/types` validator** — no new routes minted, but `just clean` before any branch-switch push stays the discipline (memory law).
