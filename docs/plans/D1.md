## Context

Prod sits **13 migrations behind** its deployed code (DB head `0005`, journal head `0018`); `/api/health` reports `migrations:"drift"` on prod. Previews appear bound to the **prod** database. The drift detector is a weak high-water-timestamp check that (per drizzle-orm #5769 ‚Äî "migrate exit codes lie") can read `ok` while the schema is actually wrong. D1 is the **"stop the bleeding"** stratum: fix **staging + code-level correctness only**, leave prod entirely untouched (prod migrate is D5). Three changes: repoint previews off prod, make deployed drift detection trustworthy (per-hash on `/api/health`), and harden the runtime client.

reopen ‚Äî ADR-0024: the `readMigrationFiles` mechanism is unchanged; only the bundler
instruction the mechanism requires is added.

## ¬ß3 ‚Äî The three changes

### Change 1 ‚Äî Repoint preview DB ‚Üí staging  *(operator env edit; after D0(b))*
- **Action:** operator sets Vercel **Preview**-scope `DATABASE_URL` to the **staging** Supabase URL (the value already in the staging custom-env / Doppler `stg` `DATABASE_URL`, ref `rwfdoqzsghqhhdapxafg`). Hand edit (no Doppler‚ÜîVercel sync exists).
- **If D0(b) shows previews already on staging:** verify-only no-op ‚Äî record and skip.
- **Verify:** redeploy any preview ‚Üí `https://<preview>/api/health` returns `db:"ok"` and (post-#2) `migrations:"ok"` (staging DB `0018` == code `0018`).
- **Rollback:** re-set Preview `DATABASE_URL` to the prior value.
- **Test:** operational probe (above) ‚Äî no unit test (env-var change).

### Change 2 ‚Äî Per-hash drift on `/api/health` + `VERCEL_GIT_COMMIT_SHA`  *(src)*
- **`src/server/health/migration-drift.ts`:** replace `dbHead === head.when` with a **per-hash multiset compare** ‚Äî `readMigrationFiles({migrationsFolder:"drizzle/migrations"})` ‚Üí set of `{hash}` vs `select hash from drizzle.__drizzle_migrations` ‚Üí `"ok"` iff the applied-hash multiset equals the journal-hash multiset; else `"drift"`; `"error"` on failure. **Pure hash-multiset compare ‚Äî no `CI`/pg_cron conditional.** Safety is *operational*: `/api/health` runs only in deployed (pg_cron, unstripped) envs, never in CI, so ADR-0022's stripped-hash false-positive cannot arise (ADR-0024 ¬ßDecision Outcome #6).
- **`src/app/api/health/route.ts`:** change the `canary` field source from `process.env.ZUGZWANG_ENV_CANARY` to `process.env.VERCEL_GIT_COMMIT_SHA` (Vercel system var, not a secret). Keep `env`, `db`, `migrations`; status-string only.
- **`scripts/check-migration-drift.ts`: UNCHANGED** ‚Äî stays timestamp+count (runs in CI against the stripped DB; per-hash would false-positive there).
- **RED-first test** ‚Äî new `tests/integration/migration-drift.integration.test.ts` (real test PG):
  1. **Planted hash mismatch (RED):** apply migrations, then `UPDATE drizzle.__drizzle_migrations SET hash='deadbeef‚Ä¶' WHERE ‚Ä¶head‚Ä¶` (fixture bypass) ‚Üí detector returns `"drift"`. The old timestamp code returns `"ok"` here ‚Üí this is the #5769 regression the rewrite closes.
  2. **In-sync:** seed `__drizzle_migrations` with **unstripped `readMigrationFiles` hashes** for the applied set ‚Üí `"ok"`. (The detector carries no pg_cron conditional; the in-sync baseline uses the committed/unstripped hashes, matching deployed reality.)
  3. **Behind:** delete the head row ‚Üí `"drift"`.
- **Rollback:** revert the PR (pure code; no data/migration touched).

### Change 3 ‚Äî `prepare:false` on the runtime client  *(src; config-only)*
- **`src/db/index.ts`:** add `prepare: false` to the `postgres(connectionString,{ max:10 })` options ‚Äî belt-and-suspenders on the `:5432` session pooler (forward-safe if a 6543 transaction pooler is ever introduced).
- **Test:** config-only ‚áí TDD-exempt (CLAUDE.md ¬ß5.6); covered by the existing integration suite + `just verify`.
- **Rollback:** remove the option (revert).
